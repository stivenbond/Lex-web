# Module Specification: FileProcessing

## 1. Overview
- **Name**: FileProcessing
- **Purpose**: Provide background asynchronous processing for media and document files uploaded to the platform, with an initial focus on PDF text extraction.
- **Schema**: `file_processing.*`
- **Primary Stakeholders**: System Services, Search Engine.

## 2. Domain Boundaries
- **Aggregate Root: ProcessingJob**: Manages the lifecycle of an individual file operation (e.g., extracting text from a PDF).
- **Entity: ProcessingStep**: Represents a specific task within a job (e.g., OCR vs. PDF Parsing).
- **Value Object: ProcessingResult**: The output metadata or text generated by the job.

## 3. Module Responsibilities
- Monitor file uploads via message bus integration.
- Orchestrate background workers for resource-intensive tasks (PDF OCR/Text Extraction).
- Track job progress and notify users/services via SignalR and Domain Events.
- Provide a standard contract for "Result Delivery" back to the originating module.

## 4. Integration & Dependencies
- **Inbound Events**: 
    - `FileUploadedEvent`: Triggered by `ObjectStorage` when a new file is finalized.
- **Outbound Events**: 
    - `FileProcessingCompletedEvent`: Carries the extracted content or link to the metadata.
    - `FileProcessingFailedEvent`: Notifies of processing errors.
- **External Dependencies**: 
    - `ObjectStorage`: For reading source bytes and potentially storing results (thumbnails/previews).
    - `Redis`: For tracking real-time job progress (SignalR).

## 5. Security & Authorization
- **Permission Constants**: `FileProcessingPermissions.cs`
- **Policies**: 
    - `CanMonitorJobs`: Internal system/admin access to the job queue.

## 6. Cross-Module Interactions
- **ObjectStorage**: Supplies the raw files.
- **Lesson/Diary**: Consumes extracted text for indexing or previewing.
- **SharedKernel**: Uses the 202 Accepted + SignalR pattern for progress reporting.
