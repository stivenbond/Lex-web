# ADR-013: Object Storage Dual-Provider Design

## Status
Accepted

## Date
2026-04-09

## Context
Lex requires a robust object storage system that can run on-premises with minimal setup. 
Two primary storage backends have been identified:
1. **PostgreSQL bytea**: Low overhead for small files, no extra infrastructure (uses existing DB).
2. **Garage**: S3-compatible, distributed storage for large files and high availability.

The system needs to support switching or utilizing both without breaking domain logic.

## Decision
Implement a dual-provider storage strategy mediated by a common adapter interface (`IObjectStorageService`).
- **Initial Mode**: Default to PostgreSQL `bytea` for ease of single-node installation.
- **Scaling Mode**: Enable Garage (S3) for larger deployments or when distributed access is needed.
- **Abstraction**: `SharedKernel` contains the adapter interfaces. Implementation details and provider-specific code reside in the `ObjectStorage` module's infrastructure.
- **Metadata**: A unified `file_records` table in the `ObjectStorage` module tracks file location (Provider, Bucket/Key, ContentType, Size) regardless of the backend.

## Migration Path
- Provide a CLI tool or background job within the `ObjectStorage` module to migrate files from `bytea` to `Garage` if an institution decides to scale up.

## Auth Model
- Files are not exposed directly. A `FileController` in the `ObjectStorage` module handles requests, verifies permissions against the requesting user's identity, and then streams the bytes from the active provider.

## Consequences
- No public S3 URLs; all access is mediated (better security, slightly more CPU overhead).
- Unified API for all other modules (Diary, Lessons, etc.) via the `ObjectStorage` contracts.
- Increased complexity in the `ObjectStorage` module to handle provider switching.
